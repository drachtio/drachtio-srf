# Dialog

The `Dialog` class represents an active, established SIP Session (typically an `INVITE` or a `SUBSCRIBE` session) spanning both UAS and UAC endpoints.

Dialog instances are returned natively via promises generated by `srf.createUAS`, `srf.createUAC`, or `srf.createB2BUA`. Once a Dialog is initialized, the application uses it to transmit subsequent in-dialog Requests or handle incoming Requests (such as `INFO`, `UPDATE`, `BYE`) targeted to the active dialog session.

## Lifecycle Methods

### `destroy(opts?: { headers?, auth? })`
Destroys the Dialog gracefully.
If the Dialog is an `INVITE`, this initiates a `BYE` Request to the remote peer. If it is a `SUBSCRIBE` Dialog, it sends a `NOTIFY` terminating the subscription state.

```typescript
// Tear down a call gracefully
await dialog.destroy();
```

### `modify(sdp?: string | opts)`
Sends an in-dialog re-INVITE (or `UPDATE` if appropriate) targeting the remote peer, typically updating the session's SDP.

- `sdp` *(string)*: The new SDP to broadcast (e.g. adding video streams to an audio-only call).

## In-Dialog Requests

To emit mid-call requests to the remote peer, use `Dialog.request` or the associated helper methods (which simply proxy their arguments into `request`). These calls automatically bind the `Call-ID`, `CSeq`, local and remote tags inherent to the active session.

### `request(opts: DialogRequestOptions): Promise<Response>`

```typescript
dialog.request({
  method: 'INFO',
  headers: {
    'Content-Type': 'application/dtmf-relay'
  },
  body: 'Signal=1\r\nDuration=200'
}).then(res => {
  console.log('INFO Request acknowledged', res.status);
});
```

## Events

You can attach listeners natively using Node.js `EventEmitter` syntax (`dialog.on('event', callback)`).

- `'destroy'`: Fired when the dialog terminates natively (usually through an incoming `BYE`).
- `'modify'`: Fired when an incoming re-INVITE contains *new* SDP configurations.
- `'refresh'`: Fired when an incoming re-INVITE simply refreshes session timers without modifying SDP.
- `'info'`: Fired on an incoming in-dialog `INFO` request.
- `'notify'`: Fired on an incoming in-dialog `NOTIFY` request.
- `'options'`: Fired on an incoming in-dialog `OPTIONS` request.
- `'update'`: Fired on an incoming in-dialog `UPDATE` request.
- `'refer'`: Fired on an incoming in-dialog `REFER` request.
- `'message'`: Fired on an incoming in-dialog `MESSAGE` request.
- `'ack'`: Fired on an incoming `ACK` validating a recent response.
- `'subscribe'`: Fired on an incoming `SUBSCRIBE` request.
- `'unsubscribe'`: Fired on an incoming un-`SUBSCRIBE` request (an `Expires: 0` header).
- `'hold'`: Fired natively when the remote party issues a re-INVITE containing a `c=0.0.0.0` or `a=sendonly` / `a=inactive` constraint.
- `'unhold'`: Fired natively when the remote party issues a re-INVITE resuming `a=sendrecv` media parameters.
